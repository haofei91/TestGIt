## 一句话总结

OpenAI 用 Codex（AI 编程智能体）从零开始建造了一个内部产品，**没有一行代码是人工写的**。5 个月、3 个工程师、100 万行代码、上线运行。这不是概念演示，是真实产品。

- **核心原则**：**不手动写代码**（No Manual Code）

## 核心理念：从"写代码"到"设计环境"

### 传统模式
```
人类 → 写代码 → 产出
```

### 新模式
```
人类（设计师）→ 描述任务 → AI 执行 → 人类审核结果
```

**人类不再写代码，而是：**
1. 设计系统架构和环境
2. 明确意图和目标
3. 构建反馈回路
4. 验证 AI 的输出

## 实际工作流程

### 日常开发流程
1. **工程师描述任务**（用自然语言）
2. **Codex 去执行**：写代码、跑测试、跑 CI、写文档
3. **Codex 自己开 PR**：不需要人类动手
4. **人类审核 PR**：可以合并，也可以让 Codex 继续改
5. **循环**：Codex 不断根据反馈修改，直到所有审核通过

### 具体例子：提交一个功能
```
你告诉 Codex：
"给用户设置页面添加双因素认证"

Codex 会自动：
1. 设计数据库结构
2. 编写后端 API
3. 编写前端界面
4. 写测试用例
5. 更新文档
6. 提交 PR
7. 自己审核自己的代码
8. 等待人类或 AI 审核反馈
9. 修改直到通过
10. 合并代码
```

## 经验教训 1：不要给 AI 一本 1000 页的说明书

### ❌ 失败做法：一个巨大的 AGENTS.md 文件
- 信息太多，AI 会"看不见"关键约束
- 所有规则都"重要"= 就没有规则重要
- 文档会过时，变成"陈旧规则的坟场"
- 难以检查和维护

### ✅ 正确做法：给 AI 一张"地图"

**核心策略：渐进式披露**

```
AGENTS.md（100 行）= 目录/地图
         ↓
docs/目录 = 详细的真实信息
         ↓
设计文档、架构文档、计划日志 = 各种专题深度内容
```

**类比**：
- ❌ 给她一本说明书 → 她淹没在细节里
- ✅ 给她一张地图 + 导航 → 她知道去哪找答案

### 知识管理结构
- **AGENTS.md**：只放最常用的导航指引（约 100 行）
- **docs/**：结构化的知识库，有索引和验证
- **架构文档**：顶层地图，展示模块和边界
- **计划日志**：记录每个功能的决策过程
- **技术债务**：集中追踪，记录在案

## 经验教训 2：让 AI 能够"感知"系统

### 问题
AI 在运行时看不到的东西 = 不存在

### 解决方案：让系统对 AI 可读

**1. 应用程序可观测**
- 日志、指标、追踪 → 对 AI 可读
- AI 可以用 LogQL 查日志
- AI 可以用 PromQL 查指标

**2. AI 可以操作 UI**
- 接入 Chrome DevTools 协议
- AI 可以截图、看 DOM、操作浏览器
- AI 可以复现 Bug、验证修复

**3. 临时测试环境**
- 每次更改，AI 可以启动独立测试实例
- 完成后自动销毁，不污染主环境

### 具体例子
```
"确保服务启动在 800ms 内完成"
"这四个关键用户旅程的响应时间不超过 2 秒"

→ AI 可以直接查指标，验证这些条件
```

## 经验教训 3：架构边界要严，细节要松

### ❌ 不要：规定"怎么做"

### ✅ 要：规定"不能做什么"

**严格边界示例**（每个业务域）：
```
Types → Config → Repo → Service → Runtime → UI
         ↓           ↓        ↓        ↓
      只能向前依赖（单向数据流）
```

**具体规则**：
- 每个业务域只能"向前"依赖，不能逆向
- 跨领域（认证、日志、监控）通过固定接口接入
- 违反规则 → linter 自动拦截，PR 被打回

### 类比
就像城市的交通规则：
- ✅ 可以在路口左转
- ❌ 不能逆行
- 规则清楚，AI 可以自由驾驶

## 经验教训 4：代码质量靠"黄金原则"

### 问题
AI 会复制现有代码库的模式，包括**不理想的模式**

### 解决方案：把人类品味编码成规则

**示例**：
1. 优先用共享工具包，不用散落的辅助函数
2. 数据验证要严格，不搞"YOLO 式"猜测
3. 所有模式统一，有问题集中修复

### 执行机制：自动巡检机器人
- 定期扫描代码库，发现不良模式
- 自动发起 PR 修复
- 大多数修复 1 分钟内审完并合并

**类比**：像代码的"自动清洁工"，每天清理 AI 产生的"技术债务"

---

## 经验教训 5：纠错成本低，等待成本高

### 传统做法（低 AI 吞吐量）
- PR 必须所有测试通过才能合并
- 测试失败 = 阻塞

### 新做法（高 AI 吞吐量）
- **测试偶发失败 → 重跑，不阻塞**
- **纠错成本低，AI 可以快速修复**
- **等待成本高，人类注意力是稀缺资源**

### 数据对比
```
传统：等所有测试通过 → 可能等几天
新模式：先合并，出问题 AI 马上修 → 通常几分钟
```

## AI 现在能端到端做什么？

给定一个 Bug 报告，AI 现在可以：

1. ✅ 验证代码库当前状态
2. ✅ 复现 Bug
3. ✅ 录制复现视频
4. ✅ 实施修复
5. ✅ 运行应用验证修复
6. ✅ 录制修复演示视频
7. ✅ 打开 Pull Request
8. ✅ 回应 AI 和人类的审核反馈
9. ✅ 检测并修复构建故障
10. ✅ 仅在需要人类判断时才中断
11. ✅ 自动合并代码

## 挑战与未知

### 当前挑战
1. **架构一致性**：随着代码库扩大，如何保持架构不漂移？
2. **人类判断的定位**：哪些地方必须有人类参与？
3. **系统演化**：模型能力越来越强，系统如何适应？

### 关键认知
> **构建软件仍然需要纪律，但纪律更多体现在支撑结构上，而不是代码本身。**

工具、抽象、反馈回路变得比代码本身更重要。

## 对我们的启示

| 方面 | 传统开发 | AI 优先开发 |
|------|----------|-------------|
| 人类角色 | 代码编写者 | 设计师/审核者 |

| 架构 | 后期规划 | 早期必须 |
| 文档 | 可选 | 必须（AI 需要）|
| 测试策略 | 全部通过才能合并 | 偶发失败可重跑 |
| 代码风格 | 人类审美 | 规则驱动 |
| 瓶颈 | 开发速度 | 人类注意力 |

## 总结

**不是 AI 替代人类，而是 AI 做执行，人类做设计。**

关键转变：
- 从"我来实现" → "我来描述目标"
- 从"写代码" → "建环境"
- 从"一个人干" → "一个人 + N 个 AI 干"

**明天软件开发的核心竞争力 = 设计好的环境、意图表达、反馈系统。**
