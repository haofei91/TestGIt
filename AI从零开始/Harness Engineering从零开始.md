# Harness Engineering 详细分析报告

## 摘要

Harness Engineering 不是“更会写提示词”，也不是“再包一层工作流”的小改良，而是将软件研发系统重构为 **Agent 可持续运行的工程操作系统**。其核心不在单次生成质量，而在持续交付质量：可理解、可验证、可修复、可演进。  

基于 OpenAI 官方实践，Harness Engineering 的关键是“Humans steer, agents execute”：人类负责目标、约束、验收与治理，Agent 负责实现、验证、修复与迭代。在该范式下，代码本身从“人工产物”转为“系统产物”，工程重心转向环境设计与反馈回路设计。  

> Harness 通常翻译为"驾驭者工程"，负责连接和编排各个组件，但不直接下场干活。

---

## 1. 背景与问题定义

过去三年，AI Coding 范式大致经历了三层演进：

1. **Prompt Engineering**：优化单次指令表达；核心关注是怎么写好一条指令。few-shot、chain-of-thought、角色扮演，本质上都在打磨单次输入。
2. **Context Engineering**：为模型拼接正确上下文； 单条 prompt 不够了，你得为模型动态构建整个上下文——相关文件、历史对话、工具定义、知识库检索结果——让模型做每一个决策时，都能看到它需要的信息 
3. **Harness Engineering**：设计完整运行环境与控制系统，使 Agent 长时稳定产出。

打个比方：
Prompt Engineering 是写好一封邮件。Context Engineering 是把相关附件都带上。Harness Engineering，是搭建整个工作环境——让 Agent 能持续、稳定、高质量地工作。它包含了上下文工程，但在更高的层面运作：约束、反馈循环、架构规则、工具链、生命周期管理。

## 2. Harness Engineering 的定义与边界

## 2.1 定义

Harness Engineering 是一种 Agent-first 软件工程方法：  
- 人类不直接承担主编码职责；  
- 人类负责定义目标、边界、验收标准、异常升级规则；  
- Agent 在约束系统内执行编码、测试、修复、PR 流转与部分评审闭环。

Humans steer, agents execute。 人类职责从“实现者”迁移为“系统设计者”：  
- 做需求优先级与验收定义；  
- 设计架构与边界；  
- 设计反馈循环与升级策略；  
- 在高不确定场景做最终判断。
- 
## 2.2 边界

它**不是**：
- 只改 prompt 的技巧集合；
- 只做 RAG/上下文拼接；
- 只做多 Agent 编排；
- 只追求“无人值守自动写代码”。

它**是**：
- 把“文档、规则、验证、观测、回收机制”工程化为可执行系统；
- 在吞吐上升时仍维持质量与架构一致性的治理框架。

## 3. 核心理念与机制

## 3.1 Map, not Manual（地图而非手册）

- `AGENTS.md` 做导航地图（短而稳定）；  
- 细节知识放在结构化 `docs/` 并可索引；  
- 采用 Progressive Disclosure（按需加载）减少上下文污染。

AGENTS.md 应该是大约 100 行的导航地图，告诉 Agent"去哪里找什么"，详细内容放在链接的文档里。塞太多信息反而适得其反，一切都"重要"的时候一切都不重要。Anthropic 官方博客中也有相同的论述：不仅 Skill 应当采取渐进式披露，CLAUDE.md 也应当存放引用而非手册全文。

## 3.2 嵌入而非外挂

架构规范、编码约定、常见坑点放在代码仓库的 docs/ 和 scripts/ 里，不放 Notion、不放 Confluence。存在 Slack 讨论里的技术决策、记在人脑子里的架构判断——如果不写进仓库，Agent 就像一个迟到三个月的新员工，对这些一无所知。

对 Agent 来说，运行时不可见的信息等于不存在。  因此，架构决策、约束规则、质量标准、执行计划必须沉淀在 repo 内，而不是散落在 IM、会议口头或外部文档系统。


## 3.3 机械验证而非人工检查

> 仅靠文档本身，是没法保持完全由智能体生成的代码库的连贯性的。通过强制执行不变量，而非对实施过程进行微观管理，我们令智能体能够快速交付，而且不会削弱基础。
> 在实践中，我们通过自定义的代码检查器和结构测试来强制执行这些规则，并辅以一小组“品味不变式”。例如，我们通过自定义 lint 静态地强制执行结构化日志记录、模式和类型的命名约定、文件大小限制，以及特定平台的可靠性要求。由于这些 lint 是自定义的，我们编写错误信息时会在智能体情境中注入修复指令。
> 在以人为本的工作流程中，这些规则可能会让人感到迂腐或束缚。有了智能体，它们就成了倍增器：一旦编码，它们就能立即应用于所有地方。同时，我们还明确指出了哪些地方需要限制，哪些地方不需要限制。

将分层边界、依赖方向、命名规则、文件规模等编码为机械检查。  
目标不是限制创造性，而是限制系统性退化。

代码规范写到 rules 里可能表达不清，Agent 也未必自觉遵守，但变成 lint 规则跑在验证阶段就能明确阻塞。

OpenAI 更进一步，把 Chrome DevTools Protocol 接入 Agent 运行时，让 Agent 能自己打开应用、操作 UI、截屏对比；把日志和 Metrics 暴露给 Agent，让它直接用 LogQL 和 PromQL 定位性能问题。验证不仅仅是"测试通过"，而是 Agent 能直接观察产品行为。

## 3.4 迭代自愈而非等待评审

默认流程不是“提交后等人看”，而是：
- 自动验证；
- 失败自动分析与修复；
- 限次重试；
- 超阈值再升级人工。

单次生成能力提升有上限可持续的“生成-验证-修复-再验证”闭环，才决定长期交付上限。

改完代码自动跑验证，失败了自动分析修复，1~3 轮收敛，反复失败才升级给人类。不阻塞在 Pull Request 上等人来看。这在传统开发流程里是不负责任的，但 Agentic 时代逻辑反过来了——Agent 吞吐量高、纠错成本低，反而是等待变得昂贵。他们甚至还搭建了"垃圾回收"机制：后台 Agent 定期扫描代码库，发现偏离团队规范的模式就自动开 PR 修复，像持续偿还小额技术债，而不是等它累积成大山。

## 3.5 Agent-legible Observability（对 Agent 可观测）

让 Agent 直接消费运行证据：  
- 日志、指标、trace 可查询；  
- UI 可操作（快照/截图/行为验证）；  
- 缩短“修复-确认”闭环，减少人工 QA 瓶颈。

## 3.6 Garbage Collection（持续清理）

通过周期性后台任务扫描反模式并提交修复，持续偿还小额技术债，避免集中爆发式治理。

## 3.7 不断提高的自主水平

随着越来越多的开发环节被直接编码到系统中 — 包括测试、验证、审查、反馈处理和恢复 — 该代码仓库最近跨过了一个重要门槛，使 Codex 能够端到端地驱动一个新功能。

给定一个提示，智能体现在可以：

验证代码库的当前状态
重现已报告的漏洞
录制一个演示故障的视频
实施修复措施
通过运行应用程序来验证修复
录制第二个视频，演示解决方案
打开 Pull Request
回应智能体和人类反馈
检测并修复构建故障
仅在需要判断时才交由人工处理
合并更改
此行为在很大程度上取决于此代码仓库的具体结构和工具，不应在没有类似投入的情况下假定它可以泛化 — 至少目前还不行。

## 4. 与上下文工程等概念的区别

| 维度 | Prompt Engineering | Context Engineering | Harness Engineering |
|---|---|---|---|
| 核心对象 | 单次指令质量 | 信息供给质量 | 工程系统运行质量 |
| 主要手段 | 提示词设计 | 检索与上下文编排 | 约束、验证、观测、回收、升级机制 |
| 时间尺度 | 分钟级任务 | 任务周期（天） | 长周期（周/月/季度） |
| 产出形态 | 更好回答/补丁 | 更稳任务执行 | 可持续代理化研发体系 |
| 典型失败 | 指令歧义 | 上下文缺失/污染 | 吞吐上升后质量崩塌 |

结论：  
Context Engineering 是 Harness Engineering 的必要子集，但不是充分条件。Harness Engineering 关注的是“系统闭环”，而非“上下文喂养”本身。

## 5. 落地方法论：四阶段实施路线

## Phase 1：可读（1-2 周）

目标：让 Agent “找得到、看得懂、知道下一步”。
- 建立 `AGENTS.md` 导航结构；
- 建立 `docs/` 分层目录（架构、计划、产品规则、质量基线）；
- 所有关键决策 repo 内落盘。

度量：
- 新 Agent 首次任务准备时间；
- 文档命中率（任务所需信息在 repo 内可找到比例）。

## Phase 2：可验（2-4 周）

目标：把“应该如此”变成“必须如此”。
- 标准化 test/lint/typecheck/build 命令入口；
- 引入结构化 lint 与边界测试；
- 定义失败重试与人工升级策略。

度量：
- PR 自动验证覆盖率；
- 首轮失败后自动修复成功率；
- 人工 review 阻塞时长。

## Phase 3：可观测（2-4 周，可并行）

目标：让 Agent 能基于真实行为验证质量。
- 接入日志/指标/trace 查询能力；
- 接入 UI 操作与快照能力；
- 输出“复现-修复-验证”证据链。

度量：
- Agent 独立复现缺陷比例；
- 回归缺陷逃逸率；
- 平均修复确认时长。

## Phase 4：可持续（持续）

目标：在吞吐增长中维持质量稳定。
- 建立后台垃圾回收任务；
- 建立质量评分与偏差修复节奏；
- 持续收缩重复人工劳动。

度量：
- 代码库质量分趋势；
- 反模式修复吞吐；
- 人工精力占比（实现 vs 决策）。

## 6. 常见误区与纠偏建议

1) **把 Harness Engineering 等同于“全自动”**  
纠偏：先保证“可读-可验-可观测”，再提高自治级别。  

2) **只做上下文，不做约束与验证**  
纠偏：上下文提升上限，约束与验证守住下限。  

3) **追求工具复杂度而忽视系统简洁**  
纠偏：优先最小可运行闭环，避免过早堆栈。  

4) **把规则写在文档但不落到 pipeline**  
纠偏：每条关键规范都应有对应自动检查。  

5) **把二手案例当作已证实结论**  
纠偏：建立证据分层，关键决策只用可复核证据。

## 7. 个人实践

Harness Engineering 的本质，是把 AI Coding 从“模型调用问题”升级为“工程系统设计问题”。  

它真正解决的，不是“单次写得更快”，而是“长期写得稳定且可控”。  
对团队而言，最先产生复利的投入通常不是更多 prompt，而是三件事：
- 知识结构化进 repo；  
- 验证机制机械化；  
- 反馈回路可观测且可恢复。  

当这三者成立，Agent 才会从“高波动工具”变成“可托付产能”。回到自己的项目里。Harness Engineering 的那几条原则我不仅认可，而且已经在用了——只不过形式不同、深度不同。

回到我们DX&AI的项目中，我们最终交付的是一个完整的工作流，借助这个工作量将20+个AI元件、诸多Rules+WIki、MCP工具串联起来，最终实现从PRD到代码开发到项目上线的交付，本质上就是在做Harness Engineering “负责连接和编排各个组件，但不直接下场干活”

`Rules + Wiki` 体系就是按 “Map, not Manual” 设计的：  我会把项目结构、构建命令、业务模块和组件入口写清楚，再通过链接把详细设计指向 `docs/`。编码规范、踩坑记录和执行约束沉淀在 `Agent Rules/Skills` 中，开发过程借助Spec进行记录，核心原则是“嵌入而非外挂”：关键知识尽量留在仓库内，而不是散落在外部文档平台。

我体感最明显的收益来自“验证闭环”。  我们先是把 Figma 设计稿和APP渲染 通过端口通信传给 Agent，让 Agent 能直接操控 App 做视图验证；代码每次改动后自动执行 `lint/build/check/test`。这样 Agent 不再只是“看Log日志猜问题”，而是能看到页面行为，并通过截图比对做自检和修正，这使 Agent 能够自主错误、验证修复，并直接推理 UI 的行为.这套端到端流程让产出质量从“代码能编译”升级为“功能能跑通”，尤其在夜间连续执行场景下，验收稳定性提升明显。  
回头看，`Rules + Wiki + Skill + Spec + Flux(Figma/App)` 这套组合，本质上与 Harness Engineering 的思路一致：先设计环境，再构建反馈回路。同时我们定制的SPEC流程在归档阶段进行反思来回顾更新相关AI元件，也是不断让AI系统本身在不断进化

当然，我和 OpenAI 的完整实践还有差距：
- 配置发布自动化还在开发中  
- 目前还没有把 Log 和 Trace 全量接入给 Agent 做性能定位；
- 没有形成“垃圾回收式”的自动代码库清理机制。
- 代码评审更多由 Agent 流程驱动，而不是全部下沉成自定义 lint 规则机械强制。
- APP更丰富的行为控制和检测还没实现

这些都是下一阶段要补齐的方向。

因此，我对 Harness Engineering 的判断是：它非常值得做，但不该被当成万能解法。  
在我看来，强模型决定能力上限，Harness 决定能力能否稳定落地。无论使用哪类工具，导航化知识组织、仓库内沉淀、自动化校验和持续修复，都是长期可复用的底座。  
另外，这类建设天然存在“先快后慢”的收益曲线：前期投入通常立竿见影，后期精度提升则需要更重的系统化投入。我的策略是先把规格定义与验证链条打牢，再按业务价值逐步扩大自动化覆盖面。
